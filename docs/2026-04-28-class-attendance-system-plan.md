# 班级考勤系统基础版实现计划

> 目标：在课程设计周期内优先交付一个“基础功能完整、流程可演示、结构清晰”的最小可用版本系统，并为权限控制、活体检测、抗攻击能力预留接口。

---

## 1. 实施目标

本阶段聚焦以下四个目标：

1. 完成后端最小可用版本，具备认证、学生管理、基础考勤、合照识别、情绪统计等核心接口。
2. 形成与课程评分要求一致的系统主流程，保证能够完成演示和报告编写。
3. 保留权限控制、活体检测相关接口与代码结构，便于后续继续扩展。
4. 输出进度说明文档，明确已完成内容和剩余未完成内容。

---

## 2. 基础版功能范围

### 2.1 本阶段必须完成

- 后端项目初始化与数据库建表
- 用户登录、获取当前用户信息
- 学生增删改查
- 学生人脸图片上传
- 基础考勤识别接口
- 考勤记录查询与 Excel 导出
- 合照上传与活动参与名单生成
- 活动参与频次统计
- 情绪统计接口
- 异常处理与占位安全接口

### 2.2 本阶段不做强实现

- 动作活体检测
- 纹理活体检测
- 视频攻击防御
- 教师/学生细粒度权限控制
- 真实高精度模型部署与调优
- 前端完整页面交互

## 5. 本阶段实现策略说明

### 5.1 识别服务策略

为保证快速交付，本阶段后端采用“服务层封装 + 占位实现”的方案：

- `face_service.py`：先提供基础识别逻辑与可替换接口；
- `emotion_service.py`：先返回简化情绪结果；
- `liveness_service.py`：先返回占位检测结果与说明。

这样可以先完成系统业务流程，后续再替换为真实算法模型。

### 5.2 权限控制策略

本阶段保留用户角色字段与权限依赖函数：

- 登录 Token 中保留角色信息；
- 路由层可挂载教师权限依赖；
- 实际细粒度权限暂不强制启用。

### 5.3 报告与答辩策略

在课程设计报告中，应明确区分：

- 已实现功能
- 占位接口与预留设计
- 后续优化方向

这样可以保证系统演示真实、文档表述一致、答辩逻辑清晰。
