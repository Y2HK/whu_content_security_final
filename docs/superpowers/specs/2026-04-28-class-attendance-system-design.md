> 文档类型：课程设计规格说明书

## 1. 项目概述

### 1.1 项目背景

随着人工智能技术的快速发展，人脸识别技术已广泛应用于安防、金融、教育等领域。在教育场景中，传统的点名考勤方式效率低下、耗时耗力，且存在代签等诚信问题。本课程设计以内容安全领域的人脸识别、活体检测技术为核心，结合 B/S（Browser/Server）架构，开发一套融合"基础考勤 + 合照识别 + 情绪分析"的班级考勤系统，旨在实现考勤过程的自动化、智能化，同时培养学生对计算机视觉技术、前后端交互、数据安全处理的综合工程实践能力。

### 1.2 项目目标

构建一套功能完整、安全可靠的班级考勤系统，实现以下核心目标：

1. **自动化考勤**：通过浏览器调用摄像头采集人脸信息，后端完成活体检测与人脸比对，实现无感考勤记录。
2. **合照批量识别**：支持上传班级/活动大合照，自动检测并识别其中学生身份，生成活动参与名单。
3. **情绪同步分析**：在考勤和合照识别过程中同步进行面部表情情绪分析，辅助教师了解课堂氛围。
4. **安全防护**：强化活体检测的抗伪造能力，有效抵御照片、视频等虚假攻击，杜绝虚假考勤。

### 2.2 性能需求

#### 2.2.1 识别性能指标

| 指标项 | 需求值 | 说明 |

| 活体检测响应时间 | ≤ 2s | 从人脸采集到返回活体检测结果 |

| 系统并发支持 | ≥ 10 人 | 同时在线考勤操作；SQLite WAL 模式 + FastAPI 异步可满足，若需更高并发可迁移至 PostgreSQL |

### 2.3 安全需求

#### 2.3.1 数据安全

1. **隐私保护**：系统中存储的学生人脸图像仅用于课程设计演示，不得泄露或用于其他用途；人脸特征向量采用加密存储。
2. **数据传输安全**：前端与后端通信采用 HTTPS 协议，防止数据在传输过程中被窃取或篡改。
3. **密钥管理**：人脸特征加密密钥通过环境变量 `FACE_FEATURE_KEY` 注入，课程设计场景下使用固定密钥字符串；生产环境应使用密钥管理系统（KMS）。
4. **数据备份**：支持数据库定期备份，防止数据丢失。

#### 2.3.2 身份认证安全

1. **用户认证**：教师/学生账号采用用户名+密码方式登录，密码采用 bcrypt 算法加盐哈希存储。
2. **会话管理**：使用 JWT（JSON Web Token）进行身份验证，Token 设置合理过期时间。
3. **权限隔离**：严格区分教师和学生权限，学生只能访问个人相关数据，教师可访问全班数据。

#### 2.3.3 算法安全（抗攻击能力）

1. **照片攻击防御**：动作活体要求眨眼/张嘴等动态动作，静态照片无法通过；CNN 纹理模型检测照片特有的纹理伪影和摩尔纹。
2. **视频攻击防御**：动作活体检测视频中动作的时序一致性，防止视频帧跳跃；CNN 纹理模型检测屏幕翻拍引入的频域异常。

### 2.4 其他非功能需求

| 需求类别 | 具体需求 |
|----------|----------|
| **兼容性** | 前端支持 Chrome、Edge 等主流浏览器；兼容 Windows/macOS/Linux 系统 |
| **易用性** | 界面简洁直观，操作流程不超过 3 步；提供清晰的操作提示和错误信息 |
| **可维护性** | 代码结构清晰，模块划分明确，注释规范；前后端接口使用 RESTful 风格 |
| **可扩展性** | 系统架构预留扩展接口，支持后续添加新的识别算法或功能模块 |
| **部署便捷性** | 提供一键部署脚本，依赖安装明确，可在常见开发环境中快速运行 |

**动作活体检测流程（前端本地完成）：**

动作活体检测完全在前端浏览器本地执行，无需后端 WebSocket 实时流：

```
前端浏览器（Vue 3）
  │
  ├─→ WebRTC 获取摄像头视频流
  ├─→ MediaPipe Face Mesh 实时检测 468 个关键点
  ├─→ 本地计算 EAR（眼睛纵横比）/ MAR（嘴巴开合度）
  ├─→ 验证动作时序连贯性（眨眼/张嘴）
  └─→ 动作验证通过后，采集单帧图像
      │
      └─→ 提交至后端 /attendance/check
          ├─→ 后端 CNN 纹理活体二次验证
          ├─→ ArcFace 人脸比对
          └─→ 返回考勤结果
```

**动作验证防篡改机制：**
- 前端动作检测基于 challenge_id 进行会话绑定，challenge 由后端随机生成并设置过期时间（10秒）
- 第一层防线（前端）：动作活体检测防御静态照片攻击（照片无法完成动态眨眼/张嘴动作）
- 第二层防线（后端）：CNN 纹理活体验证防御视频重放攻击，为核心安全防线
- 两层防线独立运作，前端动作验证通过后提交图像，后端纹理模型进行二次验证
- 后端记录 `action_verified` 和 `action_meta` 用于日志审计，不直接作为安全判定依据

**分层职责说明：**

| 层级 | 职责 | 技术实现 |
|------|------|----------|
| 表现层 | 用户交互、界面渲染、摄像头调用、图表展示 | Vue 3 + Element Plus + ECharts + WebRTC |
| 业务层 | 路由分发、业务逻辑编排、数据校验、权限控制 | FastAPI + Pydantic + JWT |
| 算法引擎层 | 活体检测、人脸检测/识别、情绪分析 | PyTorch/ONNX + InsightFace + DeepFace |
| 数据层 | 数据持久化、查询优化 | SQLite + SQLAlchemy |

### 3.4 算法设计

#### 3.4.1 活体检测算法（双阶段）

**第一阶段：动作活体检测**

```
输入：视频帧序列 (Frame_1, Frame_2, ..., Frame_N)
输出：动作验证结果 (True/False)

步骤：
1. 随机生成动作指令（眨眼/张嘴）
2. 使用 MediaPipe Face Mesh 实时检测面部 468 个关键点
3. 计算眼睛纵横比（EAR）和嘴巴开合度（MAR）
4. 验证动作时序：
   - 眨眼：EAR 先下降至阈值以下，再恢复
   - 张嘴：MAR 上升至阈值以上，再恢复
5. 检查动作完成时间是否在合理范围内（3-10秒）
6. 返回验证结果
```

**第二阶段：CNN 纹理活体检测**

```
输入：人脸图像 (256×256×3)
输出：活体概率 (0-1)

网络结构（轻量 CNN）：
┌─────────────────────────────────────────────────────────┐
│  Input (256×256×3)                                      │
│  ↓ Conv2D(32, 3×3) + ReLU + BatchNorm                  │
│  ↓ MaxPool(2×2)                                         │
│  ↓ Conv2D(64, 3×3) + ReLU + BatchNorm                  │
│  ↓ MaxPool(2×2)                                         │
│  ↓ Conv2D(128, 3×3) + ReLU + BatchNorm                 │
│  ↓ MaxPool(2×2)                                         │
│  ↓ Conv2D(256, 3×3) + ReLU + BatchNorm                 │
│  ↓ GlobalAveragePooling2D                               │
│  ↓ Dense(128) + ReLU + Dropout(0.5)                    │
│  ↓ Dense(1) + Sigmoid                                   │
└─────────────────────────────────────────────────────────┘

训练策略：
- 数据集：优先使用 CASIA-FASD / OULU-NPU 公开数据集进行训练和基准测试；如时间充裕，可补充自建样本提升模型泛化能力
- 损失函数：Binary Cross-Entropy
- 数据增强：随机裁剪、颜色抖动、高斯模糊
- 关键特征：分析面部纹理的细粒度差异、频域伪影

**Fallback 方案与切换策略：**
- 运行时自动降级策略：启动时优先加载自研模型（`models/custom_live.pth`），若文件不存在或加载失败则自动降级到 CDCN 预训练权重
- 主 fallback：CDCN（Central Difference Convolutional Networks）在 CASIA-FASD 上预训练的权重，输入尺寸 256×256，直接加载推理
- 次 fallback：集成开源 Silent-Face-Anti-Spoofing 库（输入尺寸 80×80），提供基础活体检测能力；算法引擎层统一封装输入预处理（resize + normalize），对外暴露统一接口
- 自研模型作为加分项展示，fallback 方案保证系统可用性；Phase 3 交付标准为 fallback 方案可用即可
```

#### 3.4.2 人脸识别算法

```
输入：人脸图像
输出：人脸特征向量 (512维) / 匹配身份

流程：
1. 人脸检测：RetinaFace
   - 检测图像中所有人脸位置及关键点
   - 输出：边界框 + 5个面部关键点

2. 人脸对齐：
   - 根据检测到的关键点进行仿射变换
   - 将人脸对齐到标准模板（112×112）

3. 特征提取：ArcFace (ResNet-50/100)
   - 输入对齐后的人脸图像
   - 输出 512 维归一化特征向量

4. 人脸比对：
   - 计算待识别人脸与库中特征的余弦相似度
   - 相似度 > 阈值（默认 0.6，可配置）则判定为同一人；合照场景建议阈值 0.55 以提高召回率
- 合照场景下，情绪分析复用 RetinaFace 的检测结果，避免重复人脸检测，提升推理效率
- 相似度 ≤ 阈值时标记为 "unknown"，不纳入活动参与名单，但保留在原始检测日志中供人工复核
- 阈值由后端 `config.py` 统一配置，教师不可调；合照和单人场景使用不同默认阈值（合照 0.55，单人 0.6）
   - 返回最匹配的学号和姓名
```

#### 3.4.3 情绪分析算法

```
输入：人脸图像
输出：情绪类型及置信度

流程：
1. 使用 DeepFace 框架分析
2. 内置模型：FER2013 训练的 CNN 或 VGG-Face
3. 输出 7 种情绪的概率分布：
   - angry（愤怒）、disgust（厌恶）、fear（恐惧）
   - happy（开心）、sad（悲伤）、surprise（惊讶）、neutral（平静）
4. 取概率最高者作为情绪分类结果
```

## 6. 风险评估与应对

| 风险项 | 风险描述 | 应对措施 |
|--------|----------|----------|
| **模型精度不足** | 自研活体检测模型准确率未达标 | 提前准备备用方案（如使用开源活体检测模型）；增加数据增强策略 |
| **合照识别率低** | 大合照中人脸遮挡、模糊导致识别失败 | 设置合理的置信度阈值；对低置信度结果标注"待确认" |
| **浏览器兼容性** | WebRTC 在某些浏览器中无法调用摄像头 | 提供清晰浏览器要求说明；增加摄像头调用失败提示 |
| **性能瓶颈** | 多人合照识别响应时间过长 | 采用异步处理机制；前端增加加载状态提示 |
| **数据隐私** | 学生人脸信息泄露风险 | 本地存储，不上传云端；课程结束后清理数据 |
